踩坑与边界：什么时候 GitHub 不是最佳解

⚠️ 不适合 GitHub 的场景
敏感或合规文档
HR 档案、薪酬、合同等需要严格权限和审计的文档。
→ 留在钉钉审批 / 企业网盘 / HR 系统
实时头脑风暴
需要多人同时在线涂涂改改、快速发散的早期创意阶段。
→ 先用白板/Figma/钉钉文档跑，沉淀后再进 GitHub
强 WYSIWYG 排版需求
需要复杂表格、图文混排、打印级排版的正式报告。
→ 飞书/Word 定稿后，把摘要和结论同步到 GitHub
✅ 适合 GitHub 的场景
需要版本控制的文档
PRD、RFC、技术方案、API 文档——需要多人审校、追踪变更、回滚版本。
需要共享 URL 的原型
HTML 原型、内部工具 demo、设计概念验证——需要一处修改、全员可见。
跨团队协作的交付物
PM、研发、测试、设计都需要看、都需要评论的产出物。
